\hypertarget{README.md_autotoc_md44}{}\doxysubsection{\texorpdfstring{When Parsing an i\+Cal Feed}{When Parsing an i\+Cal Feed}}\label{README.md_autotoc_md44}
Parsing \href{https://en.wikipedia.org/wiki/ICalendar}{\texttt{ i\+Cal/i\+Calendar/\+ICS}} resources can pose several challenges. One challenge is that the specification is a moving target; the original RFC has only been updated four times in ten years. The other challenge is that vendors were both liberal (read\+: creative) in interpreting the specification and productive implementing proprietary extensions.

However, what impedes efficient parsing most directly are recurrence rules for events. This library parses the original calendar into an easy to work with memory model. This requires that each recurring event is expanded or exploded. Hence, a single event that occurs daily will generate a new event instance for each day as this parser processes the calendar (\`{}\$default\+Span\`{} limits this). To get an idea how this is done take a look at the \href{https://user-images.githubusercontent.com/624195/45904641-f3cd0a80-bded-11e8-925f-7bcee04b8575.png}{\texttt{ call graph}}.

As a consequence the {\itshape entire} calendar is parsed line-\/by-\/line, and thus loaded into memory, first. As you can imagine large calendars tend to get huge when exploded i.\+e. with all their recurrence rules evaluated. This is exacerbated when old calendars do not remove past events as they get fatter and fatter every year.

This limitation is particularly painful if you only need a window into the original calendar. It seems wasteful to parse the entire fully exploded calendar into memory if you later are going to call the \`{}events\+From\+Interval()\`{} or \`{}events\+From\+Range()\`{} on it.

In late 2018 \href{https://github.com/u01jmg3/ics-parser/pull/190}{\texttt{ \#190}} added the option to drop all events outside a given range very early in the parsing process at the cost of some precision (time zone calculations are not calculated at that point). This massively reduces the total time for parsing a calendar. The same goes for memory consumption. The precondition is that you know upfront that you don\textquotesingle{}t care about events outside a given range.

Let\textquotesingle{}s say you are only interested in events from yesterday, today and tomorrow. To compensate for the fact that the tricky time zone transformations and calculations have not been executed yet by the time the parser has to decide whether to keep or drop an event you can set it to filter for {\bfseries{+-\/2d}} instead of +-\/1d. Once it is done you would then call {\ttfamily events\+From\+Range()} with +-\/1d to get precisely the events in the window you are interested in. That is what the variables \`{}\$filter\+Days\+Before\`{} and \`{}\$filter\+Days\+After\`{} are for.

In Q1 2019 \href{https://github.com/u01jmg3/ics-parser/pull/213}{\texttt{ \#213}} further improved the performance by immediately dropping {\itshape non-\/recurring} events once parsed if they are outside that fuzzy window. This greatly reduces the maximum memory consumption for large calendars. PHP by default does not allocate more than 128MB heap and would otherwise crash with {\ttfamily Fatal error\+: Allowed memory size of 134217728 bytes exhausted}. It goes without saying that recurring events first need to be evaluated before non-\/fitting events can be dropped.
